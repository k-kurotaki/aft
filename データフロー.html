主要なエンティティ間関係

顧客マスタ 1 — N POヘッダ：1顧客は複数POを発行する。
POヘッダ 1 — N PO明細：1POは複数品目明細を持つ。
PO明細 1 — N 引当データ N — 1 在庫データ：PO明細と在庫は引当データを介してM:N。1PO明細が複数ロットから引き当てられる場合も、1ロットが複数POに分配される場合もある。
POヘッダ 1 — N 生産依頼：1POの不足分が複数の生産依頼に分割される場合あり。
生産依頼 N — 1 生産計画：複数の生産依頼が1計画ロットに集約される。
生産計画 1 — N 歩留まり実績（ロット）：1計画から複数製造ロットが生まれ、各ロットの歩留まり実績が紐付く。
POヘッダ 1 — N 出荷依頼：1POが複数回に分けて出荷される（分納）。
出荷依頼 1 — N 出荷指示：1出荷依頼が複数ロットから出荷される。
引き剥がし履歴 N — 1 顧客（元）／N — 1 顧客（先）：引き剥がし関係を双方向に追跡。
需給調整議事録 N — N PO／品目／顧客：1議事録が複数のPO・品目・顧客に影響を与え、1POも複数議事録で議論される。

第5部：データ視点で見た現状課題

課題1：データの分断（CSシステム × オーダー登録ツール）
  • PO関連データ（D1〜D6, D10, D12〜D14）はCSシステムに、生産関連データ（D3, D7, D8）はオーダー登録ツールに分散。
  • 両システム間のデータ連携が人手転記であるため、データ整合性が常に揺らぐ。
  • 同一品目・同一顧客の情報を見るのに2システムを往復する必要があり、需給調整会議の準備工数を膨らませている。
課題2：意思決定ロジックがマスタ化されていない
  • 顧客優先度ランク（A/B/C）、引き剥がし閾値、品目間調整の段取り替えコスト評価などがマスタテーブル化されていない。
  • 結果、②⑥⑨の判断が属人的になり、トレース可能性も失われる。
  • あるべき姿：「顧客優先度マスタ」「引き剥がしポリシーマスタ」「段取り替えコストマスタ」をデータとして整備し、判断ロジックをデータドリブン化する。
課題3：歩留まりデータの活用不全
  • 歩留まり実績（D9）は蓄積されているが、原料ロットマスタ・製造条件との紐付が弱く、予測モデル化の素材として使えていない。
  • 結果、生産計画策定（⑤）と納期一次回答（④）が経験則ベースとなり、二次回答との乖離が常態化。
  • あるべき姿：歩留まり実績データに製造条件・原料ロット・季節性を構造化して紐付け、予測AIの学習基盤とする。
課題4：ログ系データ（議事録・引き剥がし履歴）の構造化不足
  • 需給調整議事録（D11）と引き剥がし履歴（D13）は本来は最重要の意思決定ログだが、フリーテキストもしくは属人的メモにとどまる。
  • 過去の引き剥がし発生パターン・特定顧客への連続引き剥がし・調整判断の妥当性検証が事後的にできない。
  • あるべき姿：構造化スキーマ（影響顧客・品目・数量・連鎖影響・補填結果）でログを取得し、後続のAI学習・意思決定支援に活用する。
課題5：リアルタイム性の欠如
  • 在庫データ（D5）の更新タイミングが拠点・品目によりばらつき、PO受領時引き当て（②）と出荷時引き当て（⑨）でズレが発生 → 引き剥がしの根本原因の一部。
  • 生産計画データ（D8）はExcel運用のため、版管理・変更履歴・他システムへの伝播がリアルタイムでない。
  • あるべき姿：在庫・計画・引当・歩留まり予測を単一のSSOT（Single Source of Truth）に集約し、イベント駆動でリアルタイム伝播させる。
課題6：トレーサビリティ・データ系譜の欠落
  • 「PO → 引当 → 生産依頼 → 生産計画 → 製造ロット → 歩留まり → 出荷指示 → 出荷実績」というサプライチェーン全体のデータ系譜（Lineage）が分断されている。
  • 顧客から「この出荷品はいつ・どの計画・どの原料ロットから生まれたか」を即答できない。
  • あるべき姿：マスタID・トランザクションIDの一貫性を担保し、End-to-Endトレーサビリティを実現する。これは半導体業界の品質要求（不具合トレース）への対応としても必須。
